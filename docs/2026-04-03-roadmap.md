# Housing Calculator — Product Roadmap

## Dynamic, market-aware evolution

---

## Purpose

This document describes a staged path from the current v1 calculator toward a more dynamic, market-aware tool.

The intent is to increase usefulness without committing prematurely to data-pipeline complexity.

**Guiding principle:** evolve the application incrementally rather than leaping from a static calculator to a full listing-ingestion product.

---

## Current state (v1)

The existing application already provides:

- calculation of true monthly ownership cost
- monthly ownership premium versus rent
- carry classification for the property
- comparison of own versus rent-plus-invest over time

These capabilities define a coherent analytical lens. Subsequent versions should preserve this core.

---

## Product direction

### v1 — Static calculator

**Question addressed:** Am I overpaying to own this house relative to renting an equivalent property?

### v2 — Market-aware calculator

**Question addressed:** Am I overpaying to own this house relative to rents and assumptions typical of this market?

### v3 — Property-aware calculator

**Question addressed:** Am I overpaying to own this specific listing relative to local rent and market conditions?

### v4 — Workflow tool / decision engine

**Question addressed:** Which listings or neighborhoods merit consideration under my constraints?

---

## Rationale for staged delivery

Advancing directly to live listings entails:

- data sourcing and provenance
- schema design and evolution
- location normalization
- address matching
- ongoing ingestion maintenance
- legal and licensing considerations
- scope expansion beyond the current product

That scope is materially different from the present tool. A staged approach keeps the product understandable and allows each layer to demonstrate value before the next is added.

# Version roadmap

---

## V1.5 — Hardening and clarification

### Objective

Ensure the current calculator is reliable, maintainable, and suitable as a foundation for later work.

### Target outcomes

- code organization and refactoring as needed
- automated tests for core model logic
- README aligned with public use
- AGENTS.md (or equivalent) for contributors and automated tooling
- clearer language and UI hierarchy

### Product focus

Remains a static calculator, with improved trustworthiness and extensibility.

### Rationale

Reduces technical debt before introducing external data dependencies.

### Candidate enhancements

- improved explanatory copy
- tighter distinction between “true cost” and equity conversion in labels and help text
- preset scenarios, for example:
  - low-rate environment
  - baseline / current environment
  - price-correction scenario

---

## V2 — Market-aware calculator

### Objective

Add geographic context without requiring live listing integration.

### Concept

Rather than requiring users to supply all rent assumptions manually, preload market-level defaults.

### Planned capabilities

- user selection of market, metro, or ZIP-level proxy
- default rent assumptions populated for the selected market
- optional defaults for appreciation and rent-growth assumptions
- comparison of the entered property against local benchmarks

### Minimum viable dynamic inputs

- typical local rent level
- rent growth assumption
- optional benchmark rent-to-price ratio

### Expected impact

Usability improves when the rent side of the model is anchored to market norms rather than invented from scratch.

### Rationale for this sequencing

Grounds the model in observable market context without listing ingestion.

---

## V2 data strategy

### Preferred initial approach

Use a downloadable dataset as a versioned input artifact.

Workflow characteristics:

- obtain data as CSV (or similar)
- normalize and clean in a repeatable pipeline
- load into a simple local table or lightweight store
- document the refresh and versioning process

### Advantages

- straightforward to reproduce
- low operational overhead
- no runtime API dependency for v2

### Example workflow

1. obtain the latest rent (or rent-proxy) dataset
2. run a normalization script
3. persist cleaned output in a structured form
4. map user-selected market to stored records
5. surface default assumptions in the UI

### Design constraint

At this stage, data should support **defaults**, **benchmarks**, and **context**—not imply listing-level precision.

---

## V3 — Property-aware calculator

### Objective

Progress from market-aware defaults to listing-oriented workflows.

### User inputs

One or more of:

- address
- listing URL
- manually entered property attributes

### System behavior

- populate core fields automatically where feasible
- estimate rent context from market and location
- run the own-versus-rent analysis with those inputs

### Product outcome

Primary value proposition: **paste a listing (or equivalent) and receive an immediate sanity check**—a stronger public-facing offering than market-only defaults.

---

## V3 — Implementation options

### Path A — Manual listing entry first

User supplies price, taxes, HOA (if applicable), insurance and maintenance estimates, and location; the application applies local rent defaults.

**Strengths:** low complexity; no scraping or third-party listing API required; high practical utility.

### Path B — Address-first workflow

User provides an address; the system infers location, local rent benchmarks, and optionally tax estimates where data permits.

**Challenges:** address normalization, geocoding, and broader data integration.

### Path C — Listing URL ingestion

User provides a listing URL; the system extracts or maps fields automatically.

**Challenges:** data access, schema stability, fragility, and increased legal and maintenance exposure.

### Recommendation

For v3, prioritize **manual property entry combined with local rent defaults**. This captures most of the user value at substantially lower complexity than full ingestion.

---

## V4 — Screening / decision engine

### Objective

Support evaluation of multiple properties or neighborhoods within a single workflow.

### Candidate capabilities

- comparison across multiple candidate listings
- sorting by ownership premium
- ranking by carry regime
- break-even hold period
- optionality and liquidity framing where applicable
- neighborhood-level scanning (where data supports it)

### Product positioning

The product shifts from **calculator** toward **housing decision support**.

### Strategic note

This phase is where differentiation and extensibility become most pronounced.

# Data architecture

---

## Summary

Do not default to an API unless it is clearly simpler than a CSV-based ingest path.

For near-term delivery, **downloaded data, lightweight normalization, and a documented refresh process** are likely sufficient.

---

## V2 architecture recommendation

### Inputs

- user-entered property assumptions
- local rent context from a stored market dataset

### Storage options

- static JSON derived from CSV
- SQLite or comparable lightweight store
- PostgreSQL only where already justified by the deployment environment

### Recommendation

For v2, prefer the simplest path: **CSV → cleaning script → normalized JSON or a small table**.

Use heavier database infrastructure only when it materially simplifies operations; avoid complexity for its own sake.

### Principle

Adopt the lightest storage layer that keeps refresh and lookup straightforward.

---

## Illustrative schema for market rent data

A fuller starting schema might include:

- `region_id`
- `region_name`
- `state`
- `metro_name`
- `date`
- `observed_rent_index`
- `estimated_equivalent_rent`
- `source`
- `last_updated`

A minimal v2 snapshot might use:

- `region_name`
- `state`
- `metro_name`
- `current_rent_benchmark`
- `rent_growth_assumption`
- `source`
- `last_updated`

The minimal variant suffices when the product only needs current default values.

---

## Data refresh workflow

Keep the process explicit and repeatable.

Example:

1. download the latest source file
2. run the normalization script
3. validate a sample of the output
4. update the application data artifact or table
5. record the refresh date in README or metadata

This level of discipline is adequate for v2.

# Questions addressed by version

---

## V1 / V1.5

What is the ownership premium for this **hypothetical** property?

## V2

What is the ownership premium in this **market**?

## V3

What is the ownership premium for this **specific** property?

## V4

Which properties or neighborhoods merit **evaluation** at all?

This sequence is cumulative: each stage builds on the prior.

---

# Deferred scope

The following are intentionally out of scope for early phases:

- full MLS-style browsing
- complex geospatial search
- scraping-heavy architectures
- overfitted forecasting models
- user accounts and authentication
- saved dashboards and personalization
- deep tax-optimization features
- large administrative surfaces for data updates

These may be reconsidered once the core use case is validated in production.

---

# Suggested build order

## Immediate priorities

1. complete refactoring and hardening of the repository
2. merge stabilization changes via the normal review process
3. collect qualitative feedback on the current tool

## Subsequent development cycle

4. add market presets and scenario defaults
5. select a lightweight data ingestion approach
6. integrate market rent context into the UI and model

## Later phases

7. support manual, property-specific workflows
8. evaluate address or listing ingestion only if benefits clearly outweigh cost and risk

---

# Scope filter for future work

**Criterion:** Does this change make the core judgment faster and clearer?

If yes, it aligns with strategy. If not, it is likely scope creep.

The central question the product should continue to answer:

**Am I paying too much to own this house relative to renting and preserving optionality?**

That question remains the backbone of the product.
